## 2. `KUNDEN` — DDL (as extracted)

```sql
-- ATLAS: public.KUNDEN  (PostgreSQL)
-- System of record for customers since 1998.
CREATE TABLE kunden (
  kd_nr        INTEGER       NOT NULL,             -- customer number; PK
  anrede       CHAR(1),                            -- salutation code
  name1        VARCHAR(60),                         -- see profiling: overloaded
  name2        VARCHAR(60),
  vkz          CHAR(1),                             -- F = Firma, P = Privat
  email        VARCHAR(120),
  telefon      VARCHAR(40),
  geb_datum    DATE,                                -- date of birth
  werbung_ok   CHAR(1)       DEFAULT 'N',           -- marketing consent
  angelegt_am  TIMESTAMP     NOT NULL DEFAULT now(),
  CONSTRAINT pk_kunden PRIMARY KEY (kd_nr)
);
-- No CHECK constraints on anrede / vkz / werbung_ok despite their coded nature.
-- No NOT NULL on email/telefon/name1/name2.
```

**Structural facts the agent can take as ground truth (extracted, not inferred):**

- `kd_nr` is the primary key, `INTEGER`, `NOT NULL`.
- `angelegt_am` is `NOT NULL`.
- All coded columns (`anrede`, `vkz`, `werbung_ok`) are **un-constrained
  `CHAR(1)`** at the DB level — the "enum" is convention, not enforcement. This
  is why profiling (below) is needed to discover the *actual* value sets.

→ In the spec these become exact types: `KD_NR INT (pk, required)`,
`ANGELEGT_AM TIMESTAMP (required)`, etc.

## 3. Column profiling (aggregates only)

> Generated by the deterministic profiler. Counts are over **331,408** rows
> (current live count; numbering runs higher due to the 2009 archive purge —
> see §5 Q3). Only distributions and counts crossed the network boundary; no
> individual values for PII columns are reproduced here beyond format exemplars.

| Column | Distinct | Null % | Notable distribution / finding |
|---|---:|---:|---|
| `KD_NR` | 331,408 | 0% | Unique. Max value ≫ row count → numbering gaps. |
| `ANREDE` | 4 | 1.2% | `H` 61%, `F` 38%, `D` 0.6%, **`X` 312 rows (0.09%)** |
| `NAME1` | ~330k | 0.1% | Free text. **Overloaded**: holds company name when `VKZ='F'`, surname when `VKZ='P'`. |
| `NAME2` | ~250k | 24% | Free text; first name for persons, address line for some companies. |
| `VKZ` | 2 | 0.0% | `F` (Firma) 72%, `P` (Privat) 28%. |
| `EMAIL` | ~300k | 8% | **4.1% fail RFC 5322** — predominantly trailing `;` (2011 import artifact). |
| `TELEFON` | ~310k | 11% | **19 distinct format shapes** observed (see §4). Free text. |
| `GEB_DATUM` | ~40k | 88% | Populated only for ~12% of rows. **Purpose unknown** (see §5 Q2). |
| `WERBUNG_OK` | 2 | 0% | `N` 82%, `J` 18%. Default `N`. |
| `ANGELEGT_AM` | — | 0% | Ranges 1998-03 → present. |

### 3.1 `ANREDE` value breakdown (the headline finding)

```
ANREDE   count      meaning (as understood)
------   --------   ----------------------------------------
H        202,159    Herr
F        125,847    Frau
D          2,090    Divers
X            312    ??? — undocumented            <-- OPEN QUESTION
(null)     1,000    no salutation
```

→ Spec encodes this exactly as `ANREDE CHAR(1) (enum {H, F, D, X})` with a
`//!` warning on `X` ("appears in 312 rows and is documented nowhere") and a
`//?` question ("What does ANREDE = 'X' mean?").

> **Workshop outcome (recorded 2026-06-19, after this analysis):** sales ops
> identified `X` as a 2003 campaign code for *"Familie / Eheleute"* (addressed
> couples). That answer becomes the `X: "Familie"` line in the salutation map —
> the canonical "Gerhard's knowledge, finally written down" moment.

## 4. `TELEFON` — the 19 formats

Phone numbers are free text. The 19 observed shapes include:

```
089/12 34 56-0
+49 89 1234560
0049-89-1234560
(089) 1234 560
089 1234560
089/1234560
tel. 089 123456
0171 1234567          <- mobile
+49 171 1234567
...                   <- 10 more variants, incl. extension suffixes "-0", "-12"
```

→ Spec: `TELEFON VARCHAR(40) (pii, note "Free text. 19 observed formats incl.
'089/12 34 56-0'")`, mapped via `to_e164` with a `//!` warning that ~2% will
need a manual remediation queue. The *normalisation* is a deterministic
transform; the *residual* is a flagged wart.

## 5. Open questions (the `//?` queue)

These are questions the analysis **cannot answer from metadata** and that must
go to a human (the workshop / sales ops / the retired DBA's successor). They are
carried verbatim into the spec as `//?` so they cannot be lost.

| # | Question | Where it lands in the spec |
|---|---|---|
| Q1 | **What does `ANREDE = 'X'` mean?** 312 rows, no documentation. The v17 spreadsheet, row 214, literally says *"ask Gerhard"*. | `//?` on `ANREDE` |
| Q2 | **Why does a B2B sales system hold dates of birth (`GEB_DATUM`)?** Populated for only 12% of rows, no obvious purpose. Possible GDPR Art. 5(1)(c) data-minimisation problem. | `//?` on `GEB_DATUM`; drives the doc-06 minimisation decision |
| Q3 | **Are the `KD_NR` gaps safe to ignore?** Customer numbers are not contiguous — gaps trace to a 2009 archive purge. Confirm no live references to purged numbers. | schema `note` on `atlas_kunden` |
| Q4 | **`NAME1` overloading** — confirm the `VKZ`-driven split is exhaustive (no third case). | `note` on `NAME1` + filter in mapping |

## 6. Data-quality findings (the `//!` warnings)

Surfaced by profiling; each is attached to the field it belongs to.

| Field | Finding | Severity | Spec form |
|---|---|---|---|
| `EMAIL` | 4.1% fail RFC 5322, mostly trailing `;` from a 2011 import | warn | `//!` on `EMAIL` + `validate_email \| null_if_invalid` in mapping |
| `TELEFON` | 19 inconsistent formats; ~2% un-normalisable automatically | warn | `//!` on the `to_e164` arrow |
| `GEB_DATUM` | Personal data with `RESTRICTED` classification and unclear purpose | warn (governance) | `classification "RESTRICTED"` + see doc 06 |
| `WERBUNG_OK` | Consent flag with **inverted** target semantics (J/N → opted-out boolean) — high blast-radius if wrong | warn (governance) | commented inversion in mapping; DPO-reviewed |
